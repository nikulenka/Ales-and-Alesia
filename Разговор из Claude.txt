Каждый концепт теперь строго типизирован:

TemperatureLevel (Enum) — субъективная шкала hot/warm/cold_like/no_water. Агент никогда не предложит термометр, потому что шкала определена именно так.
Severity — серьёзность для жильца: critical/high/medium/low. Есть на Cause (что будет если это причина) и на Symptom (базовый уровень до уточнения).
FactorType — разметка факторов: primary/secondary/temporal/scope/equipment. Позволяет понять, что первичное, а что вторичное.
ObservableFactor / SecondaryFactor — два разных класса: первичный жилец сообщает сам, вторичный выявляется через eliciting_question.
Cause.prior_probability — явное поле float 0..1 с аннотацией «требует проверки специалистом». Не просто текст — машиночитаемое число.
Cause.note — поле типа «пояснительный текст / аналогия для жильца». Отдельное от description.
DiagnosisSession — структурированное состояние диагностики: какие факторы подтверждены, какой симптом подозревается, нужен ли диспетчер.
tools.py — 5 инструментов агента
update_diagnosis — агент вызывает после каждого информативного ответа жильца, фиксируя факты структурированно.
get_causes_ranked — простое байесовское обновление: prior × 1.6 за подтверждающий фактор, × 0.1 за противоречащий. Возвращает отранжированный список с severity и resolution.
get_next_question — выбирает следующий вторичный фактор, приоритизируя те, что различают топ-причины.
create_ticket — создаёт заявку с urgency (emergency/urgent/normal) и готовым сообщением для жильца.
explain_cause — возвращает description + note (аналогия) + resolution для финального объяснения жильцу.
agent.py — ReAct-граф вместо линейного

Ключевое архитектурное решение: YAML-базы знаний

Специалист ЖКУ открывает services/heating/kb/knowledge_base.yaml и работает только там — никакого Python. kb_loader.py при загрузке прогоняет YAML через Pydantic-валидацию и выдаёт понятную ошибку если что-то не так (• Поле [symptoms → 0 → possible_causes → 2 → prior_probability]: значение должно быть от 0 до 1). Флаг reviewed_by_expert: false сигнализирует что база ещё не верифицирована.

Структура платформы:

core/ — ядро, не трогается никогда. models.py с полной иерархией, base_tools.py с 5 общими tools (все принимают service: str и сами знают к какой KB обратиться через реестр), base_agent.py с одной фабрикой build_service_agent(kb) — граф один для всех четырёх сервисов.

dispatcher/agent.py — отдельный граф с temperature=0.3 (точность важнее творчества). Вызывает tool route_to_service с confidence, и при >= 0.7 передаёт управление специализированному агенту.

services/*/kb/knowledge_base.yaml — четыре файла, каждый для своего специалиста. ГВС более полный (перенесён из v1), остальные три — шаблоны с реальной структурой и пометкой reviewed_by_expert: false.

Добавить пятый сервис — создать папку, скопировать yaml, добавить одну строку в ServiceType. load_all_kbs() подхватит автоматически по glob services/*/kb/knowledge_base.yaml.
